Merge ELF into Program

Location: CodeBrowser → File → Merge ELF into Program…

Overview

This action merges an ELF binary directly into the currently open program (similar to Ghidra’s built-in “Add To Program” operation). The ELF file’s memory segments and symbols become part of the existing program’s memory map – no separate program is created.

Separate namespaces are used for the symbols of the existing binary and the newly merged binary to prevent name clashes when both share common symbol names (e.g. main, init).

Usage

  1. Open a program in the CodeBrowser.
  2. Select File → Merge ELF into Program…
  3. In the dialog that appears:
    • Click Browse… to select the .elf file to merge in.
    • Enter a Existing Binary Namespace. All global symbols currently in the open program will be moved into this namespace. The default value is the program name with its file extension removed.
    • Enter a New Binary Namespace. All global symbols loaded from the ELF will be placed into this namespace. The default value is the selected file name with the .elf extension removed.
    • Check Re-run analysis after merge if you want Ghidra to re-analyse the full merged program once the merge is complete.
  4. Click OK to start the merge.

The operation runs in the background and reports progress in Ghidra’s task dialog. It can be cancelled at any time.

What Happens

  1. A namespace with the specified existing-binary name is created in the currently open program (if it does not already exist). Every global symbol in that program is then moved into this namespace.
  2. The selected ELF file is loaded into the existing program using Ghidra’s ElfLoader. Memory segments from the ELF are added to the program’s memory map; overlays are used automatically when address ranges conflict.
  3. Every memory block that was added in step 2 is renamed to <newNamespace>:<originalName> (e.g. mylib:.text). The block comment is set to Loaded by elf-merger for traceability.
  4. A namespace with the specified new-binary name is created in the program. Every global symbol that is still in the global namespace (i.e. arrived from the ELF) is then moved into this namespace.
  5. If Re-run analysis was checked, Ghidra’s auto-analysis is scheduled for the full program.

Notes

  • Symbols that cannot be moved (e.g., because a symbol with the same name already exists in the target namespace) are skipped; a warning is written to the Ghidra application log.
  • Memory blocks that cannot be renamed are skipped with a warning.
  • The two namespace names must be different.
  • The action is only available when a program is open in the CodeBrowser.
